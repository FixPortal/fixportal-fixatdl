# FixPortal.FixAtdl 1.0 Roadmap — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.
>
> **This is a *program* plan spanning four phases.** Only **Phase 0** is specified here at bite-sized, executable fidelity, because every later phase's detail is *data-dependent* on its predecessor's output (Phase 1's test targets come from Phase 0's coverage numbers + reading the source; Phase 2's fixes need Phase 1's tests as a safety net; Phase 4's release work needs the final API surface). Per the writing-plans scope-check, each phase is its own testable subsystem and **gets its own detailed plan authored at the moment that phase begins**, saved alongside this one as `docs/superpowers/plans/2026-05-30-fixatdl-phase-N-<name>.md`. Do **not** execute a later phase from this roadmap's summary alone — write its detailed plan first.

**Goal:** Take `FixPortal.FixAtdl` from a functionally-complete-but-thinly-tested 0.1.0 port to a tested, reviewed, documented, and NuGet-published **1.0.0**.

**Architecture:** Four sequential phases gated on measurable acceptance criteria. The ordering is **tests-first**: a characterization/unit-test safety net (Phase 1) is built *before* any behaviour-changing TODO/finding remediation (Phase 2) and *before* the adversarial review (Phase 3), because every later phase mutates behaviour the FIXatdl spec contract depends on and must not regress. Release engineering (Phase 4) is last, once the surface is stable.

**Tech Stack:** .NET 10, xUnit v3 + AwesomeAssertions + NSubstitute, `coverlet.collector` (already referenced) for coverage, `dotnet-reportgenerator-globaltool` for readable coverage summaries, `dotnet-stryker` 4.14.2 (already in the tool manifest) for mutation testing, Central Package Management via `Directory.Packages.props`, GitHub Actions (`build-and-test.yml`, `mutation.yml`, `release.yml` already present).

## Locked decisions

| # | Decision | Rationale |
|---|----------|-----------|
| D1 | **Tests-first ordering** (Phase 1 before Phase 3) | On an untested codebase a review mostly reports the *absence* of tests — which we already know. Tests are also the safety net that makes Phase 2's behavioural fixes safe. Locked per user instruction "lock it". |
| D2 | **Coverage bar = ≥ 80% line / ≥ 70% branch on the core namespaces** (provisional) | Mirrors the Stryker thresholds already committed in `stryker-config.json` (`high: 80`, `low: 70`) — one consistent quality bar across line-coverage and mutation. **Finalized at the end of Phase 0** against the real baseline numbers, then written into the Phase 1 detailed plan. |
| D3 | **"Core namespaces"** = `Model.*` (Elements, Collections, Types, Controls, Reference), `Validation.*`, `Fix.*` (emission), `Xml.Serialization.*` | These hold the behaviour. Trivial empty marker/spec types (the S2094 set) and pure DTOs are *counted but not gated* — testing a marker class proves nothing. |
| D4 | **Phase 3 runs in the review worktree** `.claude\worktrees\reviewer-passes` on branch `reviewer-findings-batch5` | House rule (CLAUDE.md "Code-review pass workflow"); batch numbering is monotonic and batch 4 was already consumed by the Sonar cleanup. |
| D5 | **NuGet publish is in scope and is real work** | `release.yml` packs + creates a GitHub Release on `v*.*.*` tags but has **no `dotnet nuget push`**. Phase 4 must add the push step + `NUGET_API_KEY` secret (or confirm an alternative registry with the user). |

## Phase 1: Harden test coverage  *(detailed plan authored at phase start)*

**Goal:** Bring the core namespaces (D3) to the D2 bar with characterization + unit tests, and broaden Stryker beyond the single-file pilot.

**Scope:**
- Characterization tests for the FIXatdl model: `Model/Controls/*` (note: reflectively constructed — tests must build them the way `ElementFactory` does), `Model/Types/*` (`_t` types + their wire-value converters, the highest-value behavioural surface), `Validation/*` (edits, `StateRule_t`), and collections beyond `ParameterCollection`.
- Round-trip/emission tests for `Fix.*` output values and `Xml.Serialization.*` deserialization paths not already covered by the existing parser tests.
- Broaden `stryker-config.json` `mutate` from `**/ParameterCollection.cs` to the core namespaces; record the resulting mutation score.
- Add a coverage gate to `build-and-test.yml` so coverage can't silently regress (uses the same ReportGenerator step from Phase 0; fail the build below the D2 bar).

**Dependencies:** Phase 0 baseline doc (drives file priority order).

**Acceptance Gate:**
- Core namespaces meet the finalized D2 bar (line + branch).
- Stryker runs over the core namespaces with a recorded mutation score ≥ the configured `low` threshold (70).
- All tests green; `build-and-test.yml` enforces the coverage floor; CI green on `main`.

**Detailed-plan note:** the per-file test list, the exact characterization fixtures, and the reflectively-constructed-control test harness are written into `docs/superpowers/plans/2026-05-30-fixatdl-phase-1-test-hardening.md` **after** reading Phase 0's `Summary.txt` and the relevant source. TDD per the test-driven-development skill where the test drives a behavioural assertion; characterization-capture where we're pinning existing behaviour.
